# MS-DOS 4.0 vs. Command 64 OS — Feature Comparison & Gap Feasibility Study

**Date:** 2026-07-12
**Scope:** Compare the OSS MS-DOS v4.00 reference source (`ms-dos/v4.0`, reference-only, not part of this codebase) against the current state of `command64-os` (`src/command64/*.asm`, `src/external/*`, and `docs/*`). Identify feature gaps and assess feasibility of closing each one on 6502/C64 hardware.

This supersedes the status claims in `brain/COMMANDS.md` and `brain/EXTERNAL.md` where they've drifted from source (both files predate several recent commands and have a few stale "planned" markers — noted inline below).

---

## 1. Method

- MS-DOS side: catalogued `src/CMD/*` (36 external utility folders), `src/DOS`/`src/H`/`src/INC` (kernel), `src/DEV/*` (10 device drivers), `src/BIOS`/`src/BOOT`, `src/MEMM`/`src/MAPPER` (EMS/XMS memory management), by reading title/header comments in each folder's main source file.
- Command64 side: read `src/command64/shell.asm`'s `tableCmd` dispatch table directly (not just the user manual) to get the ground-truth list of built-in commands, plus `api.asm` (kernel service table), `vmm.asm` (REU-backed virtual memory), `apptable.asm` (loaded-program registry), `path.asm`/`file.asm` (filesystem layer), and each `src/external/*` app's source (or absence of source, for `vi`).

## 2. Current State Snapshot — Command 64 OS

### 2.1 Built-in shell commands (from `shell.asm` `tableCmd`)

`EXIT, CLS, ECHO, LOAD, DIR, VER, HELP, TYPE, COPY, DEL/ERASE, REN/RENAME, DRIVE/DEVICE/DEV, RUN/GO, SET, VOL, PATH, APPS/PS, FREE, FLUSH`

- `SET VAR=VAL` **is implemented** in source (`cmdSet` parses `VAR=VAL`, normalizes case, stores via VMM) — the user manual's "planned for a future build" note is stale.
- `PATH` sets/reads a `path` env var through the same VMM-backed SET/GET machinery — also further along than `brain/COMMANDS.md` implies ("High priority, planned").
- No `CD`/`MD`/`RD`, no `DATE`/`TIME`, no `PROMPT`, no `BREAK`/`VERIFY`.

### 2.2 External apps (`src/external/*`)

| App | Status | Notes |
|---|---|---|
| `DEBUG` | Done (v0.1.3) | Memory dump/edit/fill/compare/search, hex math, register mod, disassembler (U), disk N/L/W. Full MS-DOS DEBUG-style tool. |
| `FORMAT` | Done | Wraps 1541-family `N:name,id` via `DOS_SEND_COMMAND` kernel primitive. |
| `LABEL` | Done | Sets volume label via KERNAL command channel. |
| `CONWAY` | Done | C64-native app, no MS-DOS analog. |
| `PACMAN` | Done | C64-native app, no MS-DOS analog. |
| `DVORAK` | Done | Resident keyboard-matrix patch (TSR-style), no MS-DOS analog (closest cousin: `KEYB`). |
| `EDLIN` | Partial (Phase 2 of 3) | List/Page/Delete/Insert/EditLine/Write implemented; Quit is a bare-exit placeholder (no Y/N confirm yet — Phase 3). |
| `VI` | **Not started** | Directory contains only a `BUILD_VI` build script, no source. Planned-only. |

### 2.3 Filesystem & I/O model

- Flat, device-based filesystem — no subdirectories, no path hierarchy. `path.asm`'s `findFile` operates purely on IEC device + filename; comments explicitly confirm "no directory awareness."
- 4 supported IEC devices (8–11), switchable via `DRIVE`, with per-command target-device routing (`9:FILE`).
- File I/O goes through a `HandleTable`-backed `fileOpen/fileClose/fileRead/fileWrite` layer (`file.asm`) built on KERNAL SETLFS/SETNAM/OPEN — an 8-ish concurrent logical-file-number model, not MS-DOS FCBs or 20/255-handle tables.
- PETSCII filenames, normalized to unshifted case for matching.

### 2.4 Kernel API surface (`api.asm`, jump table at `$1000`)

15 functions: `PRINT_CHAR, PRINT_STR, OPEN_FILE, CLOSE_FILE, READ_FILE, WRITE_FILE, DELETE_FILE, RENAME_FILE, ALLOC_MEM, FREE_MEM, EXIT, PARSE_PREFIX, SEND_COMMAND, VMM_READ, VMM_WRITE`. This is roughly analogous in spirit to a minimal INT 21h — but far smaller in surface than DOS's ~90 AH functions.

### 2.5 VMM (REU-backed virtual memory)

`vmm.asm`: 4KB-page Memory Control Table at `$C000`, up to 16MB addressable via REU. Used today for env vars (SET/PATH) and the loaded-app registry (`apptable.asm`, 16-slot table). This is command64's closest analog to DOS's EMS/XMS (`MEMM`/`MAPPER`/`XMA2EMS` in the reference tree) — conceptually similar (bank-switched external RAM exposed page-wise) but purpose-built and much simpler.

### 2.6 Process/app management

`apptable.asm`: a 16-slot loaded-program registry (`APPS`/`PS` to list, `FREE` to deregister, `LOAD`/`RUN` to load+execute). This is **not** multitasking — one program runs at a time; the table just tracks what's resident in memory so slots can be reused. No MS-DOS `EXEC`/child-process/`.BAT`-call equivalent.

### 2.7 Batch/scripting

**Confirmed absent.** No `REM`, `GOTO`, `IF`, `FOR`, `CALL`, `PAUSE`, `ECHO ON/OFF`, or `.BAT`-equivalent script runner anywhere in `shell.asm`. Every command must be typed interactively (or driven externally, e.g. by a BASIC loader).

### 3.2 `COMMAND.COM` internal commands & batch processing

Internal set: `DIR, COPY, DEL/ERASE, REN, TYPE, CLS, DATE, TIME, VOL, VER, PATH, SET, PROMPT, CD/MD/RD, BREAK, VERIFY, ECHO, REM, PAUSE, GOTO, IF, FOR, CALL`, plus full `.BAT` batch-file execution: sequential command playback from a text file, `%1`–`%9` positional parameters, `GOTO :label`, conditional `IF [NOT] EXIST/ERRORLEVEL/string==string`, looping via `FOR %%x IN (...) DO`, `CALL` for batch-to-batch invocation, `PAUSE` to wait for a keypress, `REM` for comments, `ECHO ON/OFF` to suppress command echoing.

### 3.3 DOS kernel services (`src/DOS`, `src/H`, `src/INC`, `src/MAPPER`)

- FCB-based and handle-based (`OPEN`/`READ`/`WRITE`/`CLOSE`/`LSEEK`) file I/O, both coexisting for backward compatibility.
- Hierarchical directories: `CHDIR`/`MKDIR`/`F_FIRST`/`F_NEXT` (directory-entry enumeration).
- File attributes (read-only/hidden/system/archive) and timestamps (`GET_TOD`, directory entry date/time fields).
- Device drivers as first-class filesystem objects: `CON`, `AUX`, `PRN`, `CLOCK$`, `NUL`.
- I/O redirection (`<`, `>`, `>>`) and pipes (`|`, via temp files) at the command-interpreter level.
- Environment blocks (per-process copy-on-exec, inherited by children).
- Memory allocation API (`ALLOCSEG`/`FREESEG`, conventional-memory arena allocator) — analogous in *purpose* (not implementation) to command64's `DOS_ALLOC_MEM`/`DOS_FREE_MEM`.
- Country/code-page support (`GETCNTRY`, `NLSAPI`) — multi-language date/currency/collating formats.
- Critical error handler (INT 24h) — structured recovery from "drive not ready" style hardware faults, letting a program retry/abort/ignore.

### 3.4 Device drivers (`src/DEV/*`)

`ANSI` (escape-sequence console driver), `COUNTRY` (locale tables), `DISPLAY` (code-page switching for video), `DRIVER.SYS` (generic external block device), `KEYBOARD` (national keyboard layout tables), `PRINTER` (code-page printer drivers), `RAMDRIVE`/`VDISK` (RAM-backed virtual disk), `SMARTDRV` (disk cache), `XMA2EMS`/`XMAEM` (expanded-memory driver + 386 emulator).

### 3.5 BIOS/BOOT & memory managers

`BIOS`/`BOOT`: low-level sector I/O, boot-sector loader, `CONFIG.SYS` directive processing (`DEVICE=`, `BUFFERS=`, `FILES=`, `SHELL=`). `MEMM`/`MAPPER`: EMS 4.0 (LIM) and 386 virtual-86-mode extended memory management — bank-switches physical RAM pages into a "page frame," broadly the same *concept* as command64's REU-backed VMM, just far more elaborate (386 protected-mode trapping, A20 handling, multi-card support).

## 4. Gap Analysis & Feasibility Studies

Feasibility is rated for a straight 6502/KickAssembler(or ca65) port targeting the existing `command64-os` architecture (flat IEC filesystem, single-tasking, REU-backed VMM, $2E00–$CFFF user space). **High** = clear existing analog/pattern, moderate effort. **Medium** = doable but needs new subsystem or nontrivial design. **Low** = fights the hardware/architecture; large effort for small payoff. **N/A** = doesn't map to C64 hardware at all.

### 4.1 High feasibility — natural next commands

| Gap | Feasibility study |
|---|---|
| **DATE / TIME** | High. The C64 has no battery-backed RTC by default, but a CIA-based software clock (jiffy counter) or a manually-set session clock (like DOS's own software clock on machines without a RTC card) is straightforward: store hours/min/sec in zero page, tick from a CIA/raster IRQ, `DATE`/`TIME` read/print it. No persistence across power-off without extra hardware (RTC cartridge, e.g. a CMD or 1351-adjacent add-on) — document that as a known limitation, same as DOS did pre-RTC. Effort: small (~1 day), no new subsystem, mirrors existing `cmdSet`-style single command handlers. |
| **PROMPT** | High. Trivial: an env-var-backed format string (`$P$G`-style or simplified) substituted into the existing prompt-print routine. Reuses the SET/VMM env-var machinery already in place. Effort: small. |
| **MORE** | High. Pure output-pagination filter — count lines/screen rows written via `PRINT_STR`, pause on 24 lines with "-- More --", wait for keypress. No filesystem changes needed; can be built as a tiny external app that reads a file via the existing `fileOpen/fileRead` API. Effort: small (~1 day). |
| **FIND** | High. Single-file substring search using the existing `fileOpen/fileRead` handle API and a straightforward byte-scan/match loop. No new kernel primitives required. Effort: small–medium. |
| **CHKDSK (C64-flavored)** | High-Medium. A true FAT walk doesn't apply (1541/71/81 use their own BAM/directory format, not FAT), but a "read BAM, report free blocks + directory entry count + flag orphaned/40-track anomalies" utility is very buildable using `DOS_SEND_COMMAND`/status-channel reads already exercised by `FORMAT`/`LABEL`. Full `/F` repair is riskier (would need BAM-rewrite logic) — scope v1 to read-only reporting. Effort: medium (~3-5 days for read-only version). |
| **TREE (flat-list variant)** | High but low-value: since there are no subdirectories, `TREE` degenerates to `DIR` with a header. Not worth building as a distinct command — could be folded into a `DIR /W`-style option instead. Recommend skipping or merging into DIR. |
| **SORT** | Medium-High. A line-based in-memory sort filter reading stdin/a file and writing stdout/a file. Memory-bound (must fit input in available user RAM or spill to VMM pages) — for realistic C64 text file sizes (a few KB) an insertion/shell sort in RAM is fine; for anything larger, chunked merge-sort via VMM pages. Effort: medium. |

### 4.2 Medium feasibility — real but scoped subsystem work

| Gap | Feasibility study |
|---|---|
| **Batch (`.BAT`-equivalent) scripting: REM/GOTO/IF/PAUSE/ECHO ON-OFF/%1 params** | Medium. This is the single highest-value gap: DOS's automation model is entirely absent from command64 today. A minimal batch interpreter is buildable on the existing shell: read a text file into a line buffer (reuse `file.asm`), feed each line through the *existing* `shellDispatch` instead of the keyboard-read loop, add a `REM`-skip, a line-number/label index for `GOTO`, simple `IF EXIST filename`/`IF "%1"=="x"` string compares, and `%1..%9` substituted from the invoking command line before dispatch. `FOR` and nested `CALL` add real complexity (need a small call-stack in ZP or the app table's spare slots) — recommend phasing: Phase 1 = REM/ECHO/linear playback; Phase 2 = GOTO/labels; Phase 3 = IF/PAUSE; Phase 4 (optional/lower value) = FOR/CALL. Effort: medium-large (1-2 weeks phased), but reuses shellDispatch almost entirely — no new command-parsing engine needed. High payoff for a DOS-parity narrative. |
| **COMP / FC (file compare)** | Medium. Best treated as two related features, not one monolithic port. `COMP` is the natural Phase 1: a binary byte-stream compare of two files via two concurrent `DOS_OPEN_FILE` handles. The current `HandleTable` is already 8 entries / LFNs 2-9, so two input files plus normal command I/O fits the architecture. The compare loop can read both files in fixed chunks (64 or 128 bytes), compare byte-for-byte in lockstep, maintain a 24-bit logical byte offset in zero page or local app storage, and report differing offsets plus the two byte values. A 16-bit offset is acceptable only for a deliberately 1541-sized first pass; 24-bit keeps the implementation compact while covering larger 1581/SD2IEC-class files without pulling in 32-bit arithmetic and printing overhead. This should cap mismatch reporting, matching the spirit of MS-DOS `COMP`'s "stop after several differences" behavior, because printing every mismatch on a C64 would be slow and noisy. EOF asymmetry naturally reports "files are different sizes"; no seek or file-size API is required for a streaming v1. `FC /B` can share this exact backend as Phase 2, making `FC /B file1 file2` either an alias into `COMP` or a thin wrapper around a reusable compare routine. Text-mode `FC` is feasible but meaningfully larger: MS-DOS `FC` supports line compare, binary mode, case folding (`/C`), whitespace handling (`/W` and `/T`), line-number output (`/N`), abbreviated output (`/A`), and resynchronization windows (`/LBn` / `/nnnn`). On Command64, a useful text `FC` should be phased after binary compare: Phase 3 = read two files line-by-line into bounded buffers, compare lines, print line ranges and optional line numbers; Phase 4 = add `/C`, `/W`, `/T`, `/N`, and a small fixed resync window only if memory allows. Full DOS parity for code pages, path recursion, huge buffers, and elaborate resync heuristics is not worth chasing. Implementation should modularize option parsing from the start: build a small external-app argument parser that recognizes slash options, numeric option payloads, and two positional filenames, then reuse it for future commands such as `MORE`, `FIND`, `SORT`, `ATTRIB`, and `XCOPY`. Effort: small-medium for `COMP`/`FC /B`; medium-large for text `FC` with resync and options. |
| **XCOPY (multi-file copy)** | Medium. Since there's no subdirectory tree, XCOPY's recursive-tree behavior collapses to "copy all files on a device matching a wildcard pattern" — essentially a wildcard-driven loop over the existing single-file `COPY`. Needs wildcard directory matching first (see 4.3, DIR wildcard support) as a prerequisite. Effort: medium, mostly bounded by wildcard-matching work, not by XCOPY itself. |
| **ATTRIB (file attributes)** | Medium-Low. 1541/71/81 disk formats *do* have a file-type/protect-flag nibble (PRG/SEQ/USR/REL + a "locked" bit) exposed via the directory listing and settable through DOS command-channel commands (the `>` prefix trick, e.g. `S0:>FILE` isn't quite it — actual lock/unlock uses specific KERNAL-adjacent conventions). A scoped `ATTRIB +R/-R` that toggles the locked bit via `DOS_SEND_COMMAND` is plausible and would reuse the same command-channel pattern as `FORMAT`/`LABEL`. Full DOS-style hidden/system/archive bits have no CBM DOS equivalent and shouldn't be attempted. Effort: medium for the read/lock-toggle subset; the rest is N/A. |
| **MEM (memory report)** | Medium. Command64 already tracks user-program memory via `apptable.asm` (loaded-app registry with load address/size) and VMM page allocation via the MCT at `$C000`. A `MEM` command summing free vs. allocated conventional RAM (scan UserProgStart..UserProgEnd against the app table) plus REU/VMM page usage (scan the MCT) is mostly assembling data already tracked elsewhere, not new instrumentation. Effort: small-medium. |
| **PRINT (background print queue)** | Low-Medium. C64 has no interrupt-driven background job model in this OS (single-tasking, no ISR-driven scheduler for user tasks) — a literal "spooled, backgrounded while you keep working" PRINT isn't achievable without adding a task-switching primitive, which is a much bigger architectural change than the printer support itself. A synchronous "PRINT queues N files, prints them one after another, blocking" is easy (reuses the same command-channel/file-read pattern as everything else) but loses PRINT's actual defining feature (background-ness). Recommend a scoped synchronous version if wanted, and don't chase the background aspect. |

### 4.3 Low feasibility / needs a prerequisite subsystem first

| Gap | Feasibility study |
|---|---|
| **CD/MD/RD — subdirectories** | Low, and arguably **not applicable**: 1541/1571 disks have no subdirectory concept at all (flat BAM+directory, max ~144-296 entries depending on drive). 1581 and CMD/SD2IEC devices *do* support partitions/subdirectories via CBM DOS's `CD` command-channel convention (`CD:SUBDIR` / `CD//`) — so a *scoped* version restricted to 1581/SD2IEC-class devices is technically possible (send `CD:` over the command channel, same pattern as `N:` in FORMAT), but it would silently not work on a bare 1541, which is presumably command64's primary target. This needs a product decision (which drives to support) before it's an engineering question. Recommend: treat as Medium feasibility *if* SD2IEC/1581 is an explicit target, otherwise N/A. |
| **DISKCOMP / DISKCOPY (sector-level)** | Low. Requires raw sector-level GCR read/write via the 1541's own drive-side routines (job codes on the command channel, or bypassing DOS via a custom drive-code upload) — a different tier of complexity from the file-level KERNAL calls used everywhere else in this codebase. DISKCOPY in particular needs two-drive or single-drive-with-swap sector buffering logic. Feasible in principle (this is exactly what disk-copy utilities like "Disk Jockey" did) but is its own multi-week subproject, not a quick command port. |
| **SUBST / ASSIGN / JOIN (drive remapping)** | Low. These lean on DOS's drive-letter abstraction layer, which command64 doesn't have (it addresses physical IEC device numbers 8-11 directly, no logical-letter indirection table). Implementing would mean adding a full indirection layer between "device the user types" and "device actually addressed" — a real architectural addition for a command family that's low-value on a 4-device C64 setup. Recommend skipping. |
| **BACKUP / RESTORE (multi-volume spanning)** | Low. Meaningful on DOS because of small floppies vs. large hard disks needing multi-diskette spanning; on C64, disks are already the unit of storage and typical use cases are small. A basic multi-disk COPY-with-prompts is buildable (medium effort) but full BACKUP-format spanning/incremental-backup semantics is over-engineering for this platform. |
| **SHARE (file locking) / IFSFUNC / FILESYS / NLSFUNC (network redirector, code pages)** | N/A. These exist to support DOS's LAN/network-redirector and multi-language subsystems. Command64 has no networking and (per `docs/*`) is not pursuing multi-codepage text — no C64-side analog exists to hook into. |
| **APPEND (file search path for opens)** | Low. Achievable in principle by having `findFile`/`fileOpen` also probe an `append`-style env-var path list before failing — a small extension to the existing `path.asm` search logic. Low priority since most DOS-era motivation (locating data files from a working directory that isn't the program's directory) is weaker in command64's flat, wildcard-light filesystem. |
| **MODE (device configuration)** | Low-Medium, scope-dependent. A `MODE` that reconfigures the 40-column screen doesn't apply (C64 is fixed 40x25). A `MODE` that changes serial/RS-232 parameters (baud rate via the C64 User Port) is a real, narrow feature with an actual hardware target — feasible as a small scoped command if RS-232 support is ever wanted, otherwise N/A. |

### 4.4 Not applicable to C64 hardware/architecture

`FDISK`, `EXE2BIN` (no `.EXE`/relocation-header format exists here — command64's own relocatable-binary convention is a different, already-solved mechanism per `docs/programmers-reference.md`), `GRAFTABL`/`GRAPHICS` (CGA/printer graphics-mode concepts, C64 has its own fixed character ROM and no comparable printer graphics dump convention in scope), `KEYB`/`COUNTRY`/`NLSFUNC` (multi-country code pages — out of scope per project focus), `RAMDRIVE`/`VDISK`/`SMARTDRV`/`XMA2EMS`/`XMAEM` (these are DOS's *own* EMS/XMS-managing drivers — command64's VMM already fills this role directly at the OS level, so porting the DOS-side drivers would be redundant, not additive), `FASTOPEN` (directory/FAT caching — CBM DOS directory reads are already fast relative to the tiny entry counts on 1541-class media; not a bottleneck worth solving), `ANSI.SYS`-style console driver (C64's screen control is already native/direct via `CHROUT`/PETSCII control codes, no VT100-style escape layer needed), `DRIVER.SYS` (generic external block device — command64 already addresses IEC devices directly), `SHARE`/`IFSFUNC`/`FILESYS` (see 4.3).

## 5. Prioritized Recommendations

1. **Batch scripting** (§4.2) — biggest single gap vs. the "MS-DOS style command-line environment" pitch in the user manual; reuses `shellDispatch` almost entirely.
2. **DATE/TIME, PROMPT, MORE, FIND** (§4.1) — cheap wins, each closes a well-known DOS command with no new subsystem.
3. **MEM** (§4.2) — mostly surfaces data the OS already tracks (`apptable.asm`, VMM MCT).
4. **CHKDSK (read-only BAM report)** and **COMP/FC** (§4.1–4.2) — medium effort, high parity value, reuse existing command-channel and multi-handle patterns.
5. Everything in §4.3/§4.4 — defer or explicitly scope out; several (SUBST/ASSIGN/JOIN/SHARE/networking/code-pages) are architectural mismatches, not backlog items.
6. `EDLIN` Phase 3 (Quit confirmation) and `VI` remain the two known **in-progress/not-started** ports already tracked in `brain/EXTERNAL.md` — not new findings, but worth sequencing alongside the above since they're closer to done than most §4.2 items.
